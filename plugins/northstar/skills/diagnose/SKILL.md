# diagnose —— 系统性归因（缺陷入口）

回路外到达的缺陷（红因不明的测试、生产缺陷、无测试捕获的怪异行为）由此进入回路，是缺陷的标准入口。回路交接对象：**契约相**（brainstorming；新四相落地前 = 现行 architect / module 的契约部分）、**测试相**（write-test；落地前 = 现行 module 的测试部分）、**implement**；另有两个不喂回回路的出口（非缺陷关闭、外部问题留档），见「归因判定」。

## 铁律

**归因先于修复。** 未完成取证不得提出任何修复；症状修复 = 失败。"看起来像 X 就改 X"是本 skill 要消灭的行为。实验性改动（加观测 / 改跑法 / 组合复跑）允许但必须回滚——铁律禁的是未归因就落修复。

## 时机

- 测试红但原因不明（含 flaky——单跑绿全量红也是缺陷，不当环境噪声跳过）
- 生产 / 用户报告的缺陷
- 行为与预期不符但没有测试捕获
- 上一次修复没生效

越是时间紧、越是"就一个小修"，越不许跳过——系统性归因比乱试快。

## 流程（逐相推进，不得跳相）

1. **取证**：完整读错误与堆栈（行号 / 文件 / 错误码，不略过 warning——答案常就在里面）；无报错的静默失败，以日志与边界观测为取证对象。稳定复现——达标 = 拿到**可靠触发条件**（确定性最佳；确属概率性的，给出统计口径如 N 跑 M 红 + 触发条件描述亦可）；复现不了 → 继续取证，不猜。查近期变更（git diff / 新依赖 / 配置 / 环境差异）。多组件链路失败（含测试基建链路：runner 并行 / 共享库 / fixture）时，先在每个组件边界加观测（进什么 / 出什么 / 配置是否传到），跑一次定位断在哪一层，再进该层取证。
2. **模式对照**：找同库能工作的相似实现（含测试自身的实现——fixture / 隔离策略同在对照范围），逐项列差异，不放过"那个应该没关系"；照参考实现做的东西出问题，把参考完整读完再比对，不凭略读改编。
3. **假设检验**：单一假设，写下来（"我认为根因是 X，因为 Y"）；最小变更验证，一次一个变量；不成立 → 换假设，禁止在旧改动上叠加新修复。不懂就明说不懂，不装懂。
4. **归因判定（分流）**——诊断的出口不是"修好了"，是把缺陷归位（主出口唯一）：
   - **实现错**（契约已声明正确行为，实现违约；系统内并发 / 时序缺陷——缺锁 / 缺事务 / 缺幂等——属此格）→ 产出复现红测试交 implement。时序类缺陷的复现测试须把交错**钉死为确定性**（栅栏范式），不得交付概率性 flaky 测试。
   - **契约缺失 / 含糊 / 写错** → 产出契约缺口描述，回踢契约相。
   - **测试自身错**（隔离 / fixture / 断言缺陷，业务实现无辜）**或断言过弱漏网** → 产出修测 / 补测清单交测试相。测试只读纪律（CLAUDE.md §34）：修测不在 implement 内顺手做。
   - **行为符合契约、预期错位** → 非缺陷；当作新需求走契约相，或裁决"按预期工作"后关闭。与"契约缺失"分不清时不必纠结归格——两格同走契约相，按证据如实描述即可。
   - **真外部问题**（第三方 / 基础设施，系统内不可修）→ 调查记录 + 处理策略（重试 / 超时 / 监控）。95% 的"无根因"是取证不完整；系统内可修的并发缺陷不得归入此格。
   - **附带义务**：缺陷能穿网而过，通常说明测试网有洞——无论主出口是哪个，都评估是否附带补测清单。

## 收敛阀（对齐 CLAUDE.md §42）

两条阀，口径独立：
- **阀一·回路轮次**：本 skill 交接出去的修复落地后缺陷仍在，计一轮（相 3 的假设验证实验不计入此阀）。同一缺陷 ≥3 轮不收敛 → 停——这不是又一个失败的假设，是架构或契约本身错了，BLOCKED 升级给人，附各轮假设与证据，禁止第 4 轮盲试。
- **阀二·取证空耗**：连续多轮取证或换假设仍**零新证据**（被证伪但缩小了范围 = 有新证据，不算空耗）→ BLOCKED 附已集证据升级，不硬耗。

## 红旗（出现任何一条即退回相 1）

"先快修回头再查" / "改改试试" / "多处一起改跑测试" / "大概是 X" / "不完全理解但可能行" / 已试 2+ 次还想"再试一次" / 未取证先列修复清单。

## 退出

出口四选一，均产出**可交接工件**：复现红测试（→ implement）、契约缺口描述（→ 契约相）、修测 / 补测清单（→ 测试相）、外部问题调查记录 + 处理策略（留档并落防护）。修复永远发生在 implement 相，不在本 skill 内。
