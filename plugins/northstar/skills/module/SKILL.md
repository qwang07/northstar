# module —— 模块行为契约设计 + 测试派生

模块层、循环。architect 之后，对单个模块。产出模块 README + 业务意图测试（初始红）。

## 硬门
未与用户确认模块内部行为契约前，不派生测试、不交 implement。产物未过 audit 不交棒。

## 前半：模块级 brainstorm
探讨该模块内部如何实现其声明行为、边界情况、契约细节。一次一问，先提 2-3 方案及取舍再定。
已有模块：先从现有实现蒸馏隐含意图（代码是证据不是答案），与用户共同判定目标态。

## 文件结构原则
- 每文件一个清晰职责；一起变的代码放一起，按职责而非技术分层切分。
- 文件变大常是"做太多"的信号。
- 文件结构仅在 brainstorm 阶段帮助划分职责，**不写进契约 README**——文件怎么切是实现自由，写进契约 = 实现细节污染契约，违反测试范围铁律。

## 产出（守边界一：只内部不边界）
模块级 README，**只填内部行为契约，不碰边界**。模块 README = 实现契约。
与项目 README **零重叠**——边界/拓扑/接口归 architect，任何内部细节同时出现在两处 = 漂移的种子。

## 测试派生（折在本步，不单设 spec-to-test）
把模块 README 的行为契约翻译成**独立测试文件**，验证**业务意图**：

- **测试范围铁律**：以 README 声明的意图为准。README 没声明的实现选择**不测**（实现自由发挥）；唯当 README 显式把某实现约束升格为需求才测它。判据 = "README 声明了没有"，不是"这算不算实现"。
- **业务意图 vs 实现细节试金石**：把模块内部完全重写、对外行为不变，此测试还该绿吗？该绿 = 测业务意图（好）；会红 = 测实现细节（坏，把实现焊死了）。
- 测"契约边界上进去什么、出来什么"，不测"盒子里怎么搅的"。
- 下游针对边上的接口契约 I 测——与上游输出测试同指一个 I，接缝由此双向锁住。
- 一个测试一个行为；命名描述行为（名字里出现"and"→拆分）。用真实代码，非必要不 mock。
- **只写测试，不写任何实现**——实现是 implement 的领地。

## 守时间语义分层
契约里不得出现带时间戳的叙事；过程态路由出境，去向交工作流。
**试金石：本 README 能否被一份全新快照无损替换？** 不能 → 删掉。

## 回踢
若发现 architect 划定的边界/接口有误（模块无法在既定边界内自洽）→ **回踢 architect 重入**，不在模块层私自挪边界。
**收敛阀**：同一缺口回踢 architect 达阈值（默认 3 次，对齐 CLAUDE.md §33a）仍不收敛 → BLOCKED 升级给人，附该缺口回踢历史（过程态出境）。

## 退出
模块内部契约完整、测试齐备且初始为红、经用户确认 → 交 audit 审。审过 → 交 implement。
