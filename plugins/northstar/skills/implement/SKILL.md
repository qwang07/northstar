# implement —— 令红变绿（手段自选，契约自钉）

audit 通过之后：常规分支测试已全红；免测试分支验证清单已过审。本技能只补实施：令红变绿，或令清单逐项过。

## 手段按规模自选
README + 测试一旦定死，执行形态随改动规模临场决定；形态变，不变量不变：

| 改动规模 | 执行形态 |
|---|---|
| 小范围修改 | 会话内直改 |
| 整个模块 | 派发单执行者（子代理） |
| 更大 / 可并行（无共享可变状态） | 多执行者并行编排 |

三形态在本平台的具名机制与语法见**平台执行说明**（本 skill `references/` 目录，按平台取用）。

**不含对话式协调**：执行者之间互相对话与本 skill 的零上下文 + 文件交接相悖；若 implementer 之间需互相对话，说明接口契约不自足——回踢上游补契约，不用协调机制兜。

## 内循环（红 → 绿 → 重构 → 回归，逐条测试循环）
implement 不是"一把改绿"，是一个完整 TDD 小循环，逐条红测试推进：
1. **验红**：确认这条红测试因**缺功能**而红，不是笔误 / 环境错；失败原因不对 = 测试 / 契约缺陷，回踢上游，**不在此改测试**。
2. **绿**：写**最小实现**令其变绿（K2，不多做）。
3. **验绿**：跑**全套**测试，确认这条过、**未打破任何已绿**、且**输出洁净（无 error/warning）**；没满足先修再继续。
4. **重构**：保持绿清理刚写的代码（命名 / 重复 / 结构），**不加任何行为**——自己的活，不把脏绿代码交出去。
5. 重复 1–4 至本模块 scope 内全部红变绿，再过契约合规门 + 验证。

## 免测试分支（验证清单驱动）
纯一次性操作无红测试，以 audit 过审的**验证清单**为唯一验收标准：逐项执行、逐项出示判定证据（命令输出 / 可观察结果），**全部项过才算完成，禁自由心证**（"看起来成了"不是完成）。清单同测试一样只读——发现清单项错 / 不可判定 → 回踢 write-test 修清单（实施暂停），修订过审后**全部清单项重新判定**（判定命令幂等只读，重跑廉价），不做断点续证。
本分支的退出门对应适用：契约合规门照过（操作结果恰好 = README 声明的意图，无少做多做）；清单判定本身已是对真实系统的实证，验证门以此为准，不另行加验。

## 变更入口
- 纯实现的小改动（不触及契约）→ 会话内直改令其符合既有测试。
- 触及契约的改动 → **先回 brainstorming 改 README、write-test 改测试，再回此实现**，不在实现端绕过契约。

## 不变量（无论何种手段，恒成立）
1. **验收标准只读**：测试与评审发现对执行循环只读——**不得篡改、不得预判、不得软化**。唯一合法回应是「满足它」或「判定它错而回踢上游」。这一条同时罩住两种作弊：改测试迁就实现、压评审让任务通过。
2. **测试错 = 契约缺陷信号**：测试自身错回踢 write-test、契约缺失回踢 brainstorming 重新生成，绝不在循环内静默改测试打补丁。**收敛阀**：同一红测试回踢达阈值（默认 3 次）仍不收敛 → BLOCKED 升级给人，附该缺口回踢历史（过程态出境）。
3. **过程态不入契约**：中间决策、试错进 git log / Linear / 记忆，去向交工作流，**绝不渗进 README**。
4. **退出条件**：不是"测试绿了"，而是 **"测试绿了 且 契约合规门过 且 验证通过 且 契约可被全新快照无损替换"**（末项是对 README 的卫生检查：实施期间无过程态叙事渗入契约，接不变量 3，读一遍即可判）。

## 模块级退出门：契约合规（自出，不外包）
全绿后，退出前自审一刀：**实现是否恰好 = 模块 README 契约——有无少做、有无多做**。这是"测试只读"不变量的执行面，不交外部 review。
（代码质量 / 精简非本技能职责——交工作流在此门之后接精简工序 / code-review 收尾。）

## 验证（实证网，超出单测）
契约合规门过后，**跑真实程序确认系统真的工作**（不止单测绿）——怎么跑因运行时而异，交工作流接（如 `/verify`、`/run`）；implement 只声明"需实证验证"，不声明具体跑法。验证失败 = 要么实现错（修），要么测试没覆盖到的意图缺口（回踢上游补契约 + 测试，不在此私加测试）。

## 若选子代理
子代理无记忆——只看你给的输入。故：
- **输入仅 README + 测试**：零上下文子代理须能独立完成 task。不能 → 是契约没写全，回踢 brainstorming。严禁夹带指向对话历史的指令；**一个 dispatch 只描述一个 task，不粘累积历史 / 前序 task 摘要**。
- **串行派发**：绝不并行派发**有共享可变状态、会冲突的** implementer；并行仅限 task 间**无共享可变状态、且并行收益明显**时（派发前由派发者判定；对应上表"可并行 → 并行编排"，并行粒度是无冲突的独立 task，不是同一片代码的多个 implementer；以隔离机制防冲突，兑现方式见平台执行说明）。子代理提问必先答完再令其动手。
- **模型分级**：执行者的模型档位由平台绑定钉死（判断类高档 / 执行类中档 / 检索类低档），派发时不额外覆写；临时派发的选档细则见平台执行说明。
- **过程态走文件**：task 输入、报告、diff 经文件路径交接，不粘进 controller 的 context。
- **状态协议**：子代理回报 DONE / DONE_WITH_CONCERNS / NEEDS_CONTEXT / BLOCKED。处置：缺上下文→补齐再派；要更强推理→升模型重派；task 太大→拆小；计划/契约本身错→升级人（接收敛阀）。**绝不让同一模型无变更硬重试**。
- **查 diff 实证，不信自报**：子代理报"完成"≠完成——生成 diff（走文件）、派评审核对，别凭成功自报就标完成。
- **派评审不得预判**：dispatch 给评审者时禁写"别标 X""顶多 Minor""按计划就这样"——预判 = 替评审裁决、独立性失效；让它提、你在回合里裁。

## 退出
全绿（或清单逐项过）、契约合规门通过、验证通过、契约未被偷改 → 通过。回到模块循环，按项目 README 声明的**模块清单**取**下一个未完成模块**走 brainstorming（模块高度）→ write-test → audit → implement；全部模块完成则触发 brainstorming 收口（项目级退出门）。
