# write-test —— 从 README 派生可执行守护物

测试相。brainstorming 定稿 README 之后、audit 合卷之前。输入 = README（**只读**——发现契约缺口回踢 brainstorming，绝不自补、不猜）；产出 = 独立测试文件（初始红）或验证清单。**只写测试 / 清单，不写任何实现**——实现是 implement 的领地。

## 分支判据（先裁再写）

逐条契约问：**该契约需要持续守护吗**——未来改动可能再次破坏它、值得永久回归保护？
- **是** → 派生测试（业务意图测试或结构测试，见下）。
- **否**（纯一次性操作：迁移、灌数、一次性脚本——做完验完即束，无未来改动可破坏）→ 写**自定义验证清单**（见「验证清单规格」），它就是该分支的"测试"。
- 同一轮两类可并存：模块行为出测试，附带的一次性迁移出清单。**同一条声明也可拆两半**：动作本身一次性 → 入清单；动作留下的长期产物（兼容视图、新表结构、新不变量）→ 出测试。整体像"一次性任务"的 scope 里最易漏掉后一半——逐条裁，不整包贴标签。

## 测试范围铁律

- **以 README 声明的意图为准**。README 没声明的实现选择**不测**（实现自由发挥）；唯当 README 显式把某实现约束升格为需求才测它。判据 = "README 声明了没有"，不是"这算不算实现"。
- **试金石**：把模块内部完全重写、对外行为不变，此测试还该绿吗？该绿 = 测业务意图（好）；会红 = 测实现细节（坏，把实现焊死了）。
- 测"契约边界上进去什么、出来什么"，不测"盒子里怎么搅的"。
- 下游针对边上的接口契约 I 测——与上游输出测试同指一个 I，接缝由此双向锁住。
- 一个测试一个行为；命名描述行为（名字里出现"and"→ 拆分）。用真实代码，非必要不 mock。

## 结构测试范式（纯结构·工具无关）

- **架构结构测试**（输入 = 项目级 README）：声明的模块作为单元存在 + 各自声明的公共接口（接口契约 I 的形状）真实可见 / 可导入。只需语言运行时 / 文件系统，工具无关。派生时全红（实现尚不存在）→ 随各模块 implement 完成自然变绿，无需谁主动认领。模块级 README 不重复派生此类测试——接口形状已由业务意图测试自然覆盖（测试能跑 = 可导入）。
- **不验真实依赖图、不验运行时接线**：边的语义兼容已由接缝接口契约 I 在上下游行为测试中双向锁住；"A 真在运行时调用 B"属集成层，YAGNI 默认不做，真在意时由 README 显式声明系统级意图、落成集成性行为测试。
- **持续不变量也值廉价结构测试**：枚举值集完整、端点 / 路由计数、跨文档引用完整性这类"一行断言锁一类漂移"的检查——README 声明了该不变量就派生（按声明所在的 README 层落位），成本一行、回报持续。

## 强断言纪律（硬）

每条测试须**严格验证输入与输出**。以下反模式禁止（按语境类推，不限 HTTP / 数值）：
- **恒真断言**——assert true、只断"结果非空 / 不抛异常"就完事；
- **只断状态码不查 body**——200 ≠ 对，返回体的关键字段必须逐一断言；
- **子串 / 正则松匹配枚举值**——`"paid" in status` 会放过 `"unpaid"`；枚举值全等比对；
- **数值精度不锁**——金额 / 费率断言须锁定精度与舍入规则，不得浮点近似糊过。

**试金石：把实现改错一位，这条测试会红吗？** 不会红 = 零判别力，比没测更危险（制造已覆盖假象），重写断言。

## 验证清单规格（免测试分支）

- **实施前写好**——它是该操作的验收标准，先于任何实施动作存在。
- **逐条机械可判定**：每条有明确的判定命令或可观察证据，答案非"是"即"否"；"看起来对 / 大致没问题 / 抽查几条人工看看"不是清单项。判定命令应幂等只读、可重复执行。
- **强断言试金石对清单同样适用**：操作做错一位，该项会判"否"吗？不会 → 零判别力，重写（全量校验优于抽样）。
- 判定所需参照物（如迁移前基线快照）作为**前置清单项**显式声明，不靠临场补救。
- 覆盖该操作在 README 里声明的全部意图（含逆向 / 异常：失败如何发现、如何回滚）。
- 与 README 同交 audit 审；implement 以清单逐项过为完成判据。

## 初始红验证

测试派生完跑一遍：应**全红**，且红因 = 实现缺失，不是笔误 / 环境错——红因不对先修测试自身。清单分支无红可验，以 audit 审代之。

## 自检（轻量，不替代 audit）

交棒前逐条核对：每条测试 / 清单项能指回 README 的哪条声明？指不回 → 要么测试越权（删），要么 README 缺声明（回踢 brainstorming）。
**明文：本自检不替代 audit。** 作者自查带着写作记忆（知识诅咒），会用记忆补全契约缺口；零上下文独立性只有 audit 的子代理给得了。

## 回踢

README 意图含糊 / 缺失，无法派生出明确断言 → 回踢 brainstorming 补契约。**收敛阀**：同一缺口回踢达阈值（默认 3 次）仍不收敛 → BLOCKED 升级给人，附回踢历史（过程态出境）。

## 退出

测试齐备且初始红（或清单齐备且机械可判定）→ 交 audit 合卷审（README + 测试 / 清单）。审过 → 人 Gate → implement。
