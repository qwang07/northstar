# brainstorming —— 契约对话（README 唯一写入口）

契约相。回路入口：新需求直接进；缺陷经 diagnose 归因、携契约缺口进；下游（write-test / audit / implement / code-review）发现契约缺口也回踢到此。产出 = README 条款（按权威层落位：项目级 / 横切层 / 模块级）。
**全回路只有本相可写 README**——其余各相对 README 一律只读；本相只产 README——测试归 write-test，实现归 implement。

## 硬门

契约未经用户确认，不交 write-test、不写任何测试或实现代码——不论改动看起来多简单。"太简单不用设计"是反模式：未审视的假设在小活里浪费最多。契约可以短到几句话，但必须呈给用户确认。产物未过 audit 不交棒（何时审见 audit 的双检查点）。

## 目标定回合，落位定层级（高度不是入口参数）

回合以**目标**开题：这轮要达成什么、怎么算达成（成功标准）——目标即回合边界。高度是目标澄清后的**落位结果**，不是入口先验二选一：同一目标的契约条款可散落多层（如"加导出功能"= 一条模块级行为 + 一条项目级拓扑边），各写各层、互不复述。

契约条款按**权威层落位**，落到哪层用哪层骨架（见「产出」）：
- 拓扑 / 模块边界 / 接缝接口 → **项目级** README；
- 多模块共守的横切约束 → **横切层** README（落在其适用范围对应的中间层）；
- 单模块内部行为 → **模块级** README。

**粒度阀（硬）**：本轮目标需要展开**超过一个模块的内部契约**（判据机械：数本轮要新增 / 修改内部契约条款的模块数，>1 即触发。被牵连的上游模块也计入——如新拓扑边要求上游新增取数能力，上游即在数内；上游既有接口已能兑现则不在数内，问一句即可判定。给模块级 README 添加承接指针**不计入**——指针不是内部契约条款）→ 本轮止于分解——产出子目标清单 + 所需的项目级 / 横切层条款，**不得同轮顺势下钻写多个模块的内部契约**；各子目标下轮各自成轮（brainstorming → write-test → audit → implement）。"止于"限的是下钻深度不是流程：本轮产出的 README 条款仍走完 audit 契约审 → write-test（结构测试 / 持续不变量）→ audit 合卷。

## 入口判定（按模块粒度，不是项目一刀切）

逐模块问：该模块有无现实态代码？
- **无** → 直接探讨需求，凭空声明目标态。
- **有** → 先蒸馏：从 as-is 代码反推隐含 to-be，滤除过程态遗留（死路径、历史补丁、当年妥协），**并询问用户是否需要调整**。
- 半成品 = 两者按模块混合，天然处理。

起步时"模块"尚不存在：存量代码先从代码粗蒸馏出**候选模块划分**（代码是证据），零代码则由需求分解得出候选模块（见对话方法论"大问题先分解"）；再逐候选模块做上述判定。模块是职责单元，**不必与现有文件一一对应**——一文件多职责该拆、多文件一职责该并。

## 蒸馏铁律

**代码是证据，不是答案。** 绝不把"现在怎么做的"直接抄成 README——那是让现实态僭越成目标态，SSOT 出生即被污染。产出必须是人机共同判断后的目标态。
铁律禁的是**未经判断的直抄**：经蒸馏与提醒后，用户仍拍板"现状即目标态"——那已是人机共同判断的结果，照写（该裁决与理由作为过程态出境）。用户拍板保留的内容不再算"用户没要的"，YAGNI 不与用户裁决对抗。

## 对话方法论

- 一次一问，优先多选；聚焦目的 / 约束 / 成功标准。一个话题问不完就拆成多问，不一口气轰炸。
- 拿得出方案时，先提 2-3 个方案及取舍、带推荐再定——不静默替用户选择。
- 大问题先分解为子模块（各自独立、关系清晰、构建顺序），再逐模块走流程。
- **边界判据**：每个模块须能回答——它做什么、依赖什么、服务谁 / 输出什么。不读内部能否理解它？改内部会不会破坏调用方？不能 / 会 → 边界没划好。
- YAGNI 无情砍：用户没要的功能、单次使用的抽象、不可能场景的容错，从契约里删掉。

## 产出

**项目级 README**（只边界不内部）：
1. **模块清单**与各自边界接口；
2. **依赖拓扑**——每条边写成"A→B 经接口契约 I"，**边携带接口契约 I**，不只是"A 依赖 B"；
3. 刻意**不写模块内部**——探索时看到再多内部实现也忍住，那是模块级 README 的领地。

项目 README = 接口索引 + 拓扑。接口契约 I 在此声明一次（接缝由此闭合：上游输出与下游输入日后都针对同一 I 测）。

**横切层 README**（中间层，按需存在）：适用范围（罩住哪些模块）/ 共守约束（每条可检验——违反它能被观察或断言）/ 承接指针（哪些模块级契约承接它，用指针不复述）。仅当约束确实由多个模块共守时才设此层；单模块的约束直接归其模块级 README（YAGNI）。

**模块级 README**（只内部不边界）：职责与边界 / 应有功能 / 接口契约（输入 / 输出 / 错误）/ 边界情况与不变量。与上层 README **零重叠**——边界 / 拓扑 / 接缝归项目级，横切约束归其横切层，内部细节同时出现在两层 = 漂移的种子。
文件怎么切是实现自由，**不写进契约**——写进去 = 实现细节污染契约。

**一处权威 + 指针**：同一契约只在其权威层声明一次；别处需要就写指针（路径 / 条款号），**禁止跨 README 复述**——复述 = 镜像税，改一处漏一处。

## 业务流走查（定稿前必做）

把 README 契约网当状态机走查：枚举每条业务流的**正向、逆向、异常**路径，逐环节问"这一步的承接契约声明在哪"。走不通 / 无人承接（吞错闭环、信号无消费端、悬挂状态）= 契约缺口，当场补进 README。
走查范围随目标：跨模块目标走全部相关流；单模块目标走该模块参与的流即可；收口走全部流。各相各管一段，断链只有走查能抓——这是本相不可让渡的职责。

## 守时间语义分层

契约里不得出现带时间戳的叙事——决策记录 / 变更说明 / 历史背景一律出境，路由交工作流。承重约束在热文档留一 token 标记（`[承重·勿删]`），理由本身出境。
**试金石：本 README 能否被一份全新快照无损替换？** 不能 → 历史已渗入，删掉。

## 退出（路由）

契约定稿、业务流走查过、经用户确认（按目标牵涉面分两路）→
- **单模块目标**（只落了模块级条款）：直接交 **write-test**（独立测试相）派生可执行守护物（测试或验证清单——"需否持续守护"的判据归 write-test，本相不裁）；audit 留到 write-test 完成后合卷一次审。
- **跨模块目标**（落了项目级 / 横切层条款，或触发粒度阀产出拓扑分解）：先交 audit（独立评审相）审契约、省派生返工，再进 write-test（派生结构测试 / 持续不变量），另过 audit 合卷。

## 收口（项目级退出门）

全部模块经 write-test → audit → implement 完成后，回到本相收口：确认**架构结构测试**（项目级轮次由 write-test 从项目级 README 派生的那套）**全绿** + 对最终契约网**复跑业务流走查** → 项目完成（loop 终态）。

## 收敛阀

被下游反复回踢**同一缺口**达阈值（默认 3 次）仍不收敛 → BLOCKED 升级给人决策，附该缺口回踢历史（前几次各是什么缺口、怎么改的；过程态出境，进 git/Linear），不再自动重入。反复回踢同一条边常意味根因在更上游（边界本身划错），需人判断"再试"还是"上移重划"。
