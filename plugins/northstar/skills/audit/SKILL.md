# audit —— 设计产物独立评审

评审层、独立。在 architect 之后（审项目 README + 架构测试）、在 module 之后（审模块 README + 模块测试）、在 implement 之前。

## 核心机制：零上下文（硬性）
audit **必须由零上下文子代理执行**——不是"优先"，是硬性。同 session 的当前 agent 记得整个设计对话，它会用记忆补全契约缺口，正是 audit 要消灭的；故当前 agent **不得自审、不得"假装失忆"代审**。
- **平台依赖**：本 skill 依赖平台的子代理能力（虽属原生，仍是依赖）。环境无法派子代理 → audit **不可执行 → BLOCKED 升级给人**，绝不降级为当前 agent 假装失忆审。
- **dispatch 红线**：派发 audit 子代理时，prompt 里**只有 README + 测试两样东西**。controller **不得**附加任何解释、背景、项目导言或"这个项目是做什么的"。最隐蔽的失效不是假装失忆，而是 controller**好心补一句背景**、把记得的上下文偷渡给子代理——零上下文当场破功。严禁这种善意泄漏，亦严禁夹带"参考之前讨论""按说好的方式"等指向对话历史的指令。

## 为什么零上下文审 = 完整度测试
一个零上下文读者若不能仅凭 README + 测试把系统建出来，说明契约没写全——而"契约没写全"正是本框架要消灭的瓶颈。故 audit 既是评审，也是契约完整度的可执行检验，且前移到实现之前（fail fast）。

## 四审项
1. **完整度**：零上下文读者能否仅凭 README + 测试把这一层建出来？建不出 → 意图没声明全。
2. **契约↔测试一致**：测试是否忠实投影 README——没有测试断言了 README 未声明的，也没有 README 声明了却无测试覆盖的。
3. **边界卫生**：架构/模块零重叠；无时态（过程态没渗进契约）；接缝接口契约 I 在每条边上都在。
4. **内部一致**：有无相互矛盾的条款；有无能被两种读法解释的歧义。

## 判决
产出**通过 / 缺口清单**，门控交棒：
- 全过 → 放行下一相。
- 有缺口 → **回踢上游**（架构层缺口回 architect，模块层缺口回 module）由其补契约。**audit 只指出缺口，绝不自行补写契约**——自补等于评审者僭越成作者，独立性当场失效。
- **收敛阀**：同一缺口被反复回踢达阈值（默认 3 次，对齐 CLAUDE.md §33a）仍不过 → BLOCKED 升级给人，附该缺口回踢历史（过程态出境），不再自动放行/回踢。

## 退出
判决为"通过" → 交棒（architect 后交 module，module 后交 implement）。
