# audit —— 设计产物独立评审

评审层、独立。双检查点：
1. **契约审**（brainstorming 定稿 README 后）：项目级 README 必做——契约先审再派生测试，省返工；模块级及更小 scope 可省，留到合卷一次审。审 README。
2. **合卷审**（write-test 完成后，implement 之前必过）：审 README + 测试；免测试分支审 README + 验证清单（清单即该分支的"测试"，审项同样适用）。

## 核心机制：零上下文（硬性）

audit **必须由零上下文子代理执行**——不是"优先"，是硬性。同 session 的当前 agent 记得整个设计对话，它会用记忆补全契约缺口，正是 audit 要消灭的；故当前 agent **不得自审、不得"假装失忆"代审**。write-test 的轻量自检同理不算数——作者自查稀释不了知识诅咒。
- **平台依赖**：本 skill 依赖平台的子代理能力。环境无法派子代理 → audit **不可执行 → BLOCKED 升级给人**，绝不降级为当前 agent 假装失忆审。
- **dispatch 红线**：派发 audit 子代理时，prompt 里**只有 README + 测试（或验证清单）**。controller **不得**附加任何解释、背景、项目导言或"这个项目是做什么的"。最隐蔽的失效不是假装失忆，而是 controller **好心补一句背景**、把记得的上下文偷渡给子代理——零上下文当场破功。严禁这种善意泄漏，亦严禁夹带"参考之前讨论""按说好的方式"等指向对话历史的指令。**也不得预判发现**——禁写"别标 X""顶多 Minor"：预判 = 替审者裁决，独立性同样当场失效；让它提、你在回合里裁。

## 五审项

1. **完整度**：零上下文读者能否仅凭 README + 测试 / 清单把这一层建出来？建不出 → 意图没声明全。
2. **契约↔测试一致**：测试 / 清单是否忠实投影 README——没有断言了 README 未声明的，也没有 README 声明了却无覆盖的。
3. **边界卫生**：项目级 / 模块级 README 零重叠；无时态（过程态没渗进契约）；接缝接口契约 I 在每条边上都在（拓扑边只在项目级 README 在场时查，单模块材料无边则此子项不适用）；无跨 README 复述（一处权威 + 指针）。
4. **内部一致**：有无相互矛盾的条款；有无能被两种读法解释的歧义。
5. **断言强度**：测了但断言恒真 / 过弱 = 零判别力——审项 2 只查覆盖，本项查判别力。逐条问：把实现改错一位，它会红吗？恒真断言、只断状态码不查 body、松匹配枚举、精度不锁，一律计缺口。清单分支对应查：每条是否机械可判定（有判定命令 / 可观察证据，答案非"是"即"否"）。

契约审检查点只有 README 在场，审项 1、3、4 适用；2、5 留待合卷审。

## 判决

产出**通过 / 缺口清单**，门控交棒：
- 全过 → 放行下一相。
- 有缺口 → **回踢上游分流**：契约缺口（意图缺失 / 含糊 / 矛盾 / 边界不清）回 **brainstorming**；测试缺口（覆盖漏 / 断言弱 / 越界断言 / 清单不可判定）回 **write-test**。越界断言（测试断言了 README 未声明的）默认按测试缺口回 write-test 删改——README 是权威；若它暴露的其实是漏声明的真实意图，则同时报契约缺口回 brainstorming，两格分不清时并报、在人 Gate 裁。**audit 只指出缺口，绝不自行补写**——自补等于评审者僭越成作者，独立性当场失效。
- **收敛阀**：同一缺口被反复回踢达阈值（默认 3 次）仍不过 → BLOCKED 升级给人，附该缺口回踢历史（过程态出境），不再自动放行 / 回踢。

## 退出

判决为"通过" → 交棒：契约审过 → write-test；合卷审过 → 人 Gate → implement。
